# 2026-05-12 开发日志

## 1. 新增 `--use_llm` 参数（`00d5101`、`3d456c4`）

**问题**：每次运行 `translate_type_rag.py` 都要调 LLM，即使只想用 `fixed_type_map` 做快速映射。

**方案**：
- `src/java/type_resolution/translate_type_rag.py` 新增 `--use_llm` 参数，默认 `true`
- `--use_llm=false` 时跳过 LLM 调用，仅用 fixed_type_map + 自定义类型
- `--use_rag` 仅在 `--use_llm=true` 时生效
- `scripts/java/translate_types.sh` 同步新增 `use_llm` 入参（第5个参数）

## 2. 自定义类型持久化（`8ee6eab`）

**问题**：`translate_type_rag.py` 和 `create_skeleton.py` 各自维护了一份 `get_custom_types`，每次都要扫描全部 schema 文件。

**方案**：
- 删除 `translate_type_rag.py` 中本地 `get_custom_types`，统一用 `src/java/utils/get_custom_types.py`
- 新增 `save_custom_types` 函数，结果持久化到 `data/java/type_resolution/<project>/custom_types.json`
- 每次运行都会更新，`create_skeleton.py` 后续可直接读 JSON

## 3. cjc 编译验证加入自定义类型桩（`c2c9969`）

**问题**：LLM 将 `ArrayList<Student>` 正确翻译后，`is_type_loadable` 编译验证时因找不到 `Student` 定义而误判为失败。

**方案**：在 cjc 测试程序中为所有自定义类型生成空类桩定义：
```cangjie
class Student {}
class StudentManager {}
```

嵌套类取简单名（仓颉不支持嵌套类），如 `Outer.Inner` → `class Inner {}`。

## 4. 关键字/命名冲突处理重构（`b8b2df1`、`7bd7687`）

**问题**：`handle_keyword_conflicts.py` 职责混杂 —— 关键字重命名 + 结构化命名冲突 + 共享工具全在一个文件里，难以维护。

**方案**：
- 提取共享工具（`load_parser`、`extract_text_by_bytes`、`_skip_dir`、`clean_target_dirs`、`pre_scan_project`）到 `src/java/preprocessing/_shared.py`
- `handle_keyword_conflicts.py` 缩减为纯关键字冲突逻辑，调用 `_shared` 中的工具
- 新增 `handle_name_conflicts.py` 作为结构化命名冲突处理框架
- 新增 `scripts/java/handle_name_conflicts.sh` 脚本
- 关键字列表扩展：添加 `is`（`b8b2df1`）、`func` 到活跃关键字

## 5. 内部类重命名 —— extends 感知的裸引用替换（`9aae3d9`）

**问题**：Java 内部类在仓颉中需提取为顶级类，不同外部类的同名内部类会产生命名冲突；且子类通过 `extends` 继承的内部类引用需要同步重命名。

**方案**（`src/java/preprocessing/handle_name_conflicts.py`）：
- **Step 1**：预扫描收集所有顶级类名 + 每个文件的 `extends/implements` 父类集合
- **Step 2**：检测所有内部类声明
- **Step 3**：重命名为 `OuterClass_InnerClass` 格式，冲突时追加数字后缀
- **三种引用模式替换**：
  - 限定引用 `OuterClass.InnerClass`（含 import）
  - 定义文件内的裸引用
  - 子类中通过 extends 继承的裸引用
